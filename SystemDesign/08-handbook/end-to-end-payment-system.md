# Architecture Blueprint: Distributed Payments & State Reconciliation

> **Context:** Designing a highly concurrent booking/e-commerce system where we must guarantee that users are never double-charged, and that the database strictly matches the real-world financial state, even when external networks fail.

---

## 1. The Core Infrastructure & Concepts

- **Redis Distributed Lock:** Used as a fast, in-memory "bouncer" to reserve seats temporarily with a Time-To-Live (TTL) of 10 minutes.

- **Idempotency Key:** A unique identifier (usually a UUID) generated by the Client (Frontend) when the user clicks "Pay". It guarantees that if a request is retried due to a dropped connection, the backend and the payment gateway (Stripe) will recognize it as a duplicate and only process it once.

- **The Dual-Write Problem:** The danger of updating a database and publishing an event (e.g., calling a payment service) in two separate steps. If one succeeds and the other fails, the system is inconsistent.

- **Transactional Outbox & CDC (Change Data Capture):** The L5 solution to the dual-write problem.

---

## 2. The End-to-End Flow (The Happy Path)

### Seat Reservation

User selects seats. The system attempts to acquire a Redis lock for those `seat_ids` with a 10-minute TTL. If successful, the seats are temporarily reserved.

### Checkout Initiation

User clicks "Pay". The client generates an **Idempotency Key** and sends it in the payload.

### The Transactional Outbox (ACID Guarantee)

1. The API Gateway routes the request to the **Order Service**.
2. The Order Service checks the database for the Idempotency Key (enforced via a `UNIQUE` constraint).
3. Inside a **single database transaction**, the Order Service writes two things:
   - Inserts the **Order** with status `PENDING` and the Idempotency Key.
   - Inserts an **Event** payload into an **Outbox** table.

### Asynchronous Payment Trigger

1. A **CDC process** (like Debezium) monitors the Outbox table. When it sees the new event, it publishes it to a **Message Broker** (Kafka).
2. The **Payment Service** consumes the message and calls the Stripe API, passing the Idempotency Key.

### The Webhook (Push Confirmation)

1. Stripe processes the payment and fires an asynchronous **Webhook** to our system's registered Callback URL (`/webhooks/stripe`).
2. Our system receives the webhook, verifies the signature, and updates the database Order status from `PENDING` to `SUCCEEDED`.
3. The SQL row for the seat is permanently locked (`SELECT ... FOR UPDATE`) and assigned to the user.

---

## 3. The "Payment Black Hole" (Handling Unknown States)

**The Scenario:** Stripe successfully charges the credit card, but the network drops the HTTP response and/or the Webhook fails to reach our server. Our database remains stuck in `PENDING`.

**The L5 Rule:** A network timeout is an **unknown state**. You must never blindly roll back a transaction or issue a refund on an unknown state.

### The Solution: Active Reconciliation (Pull)

1. **The Cron Job:** A background worker runs every 2 minutes.
2. **The Sweep:** It queries the database for any orders that are `PENDING` AND older than 3 minutes (giving Stripe's webhooks a grace period to arrive).
3. **The Verification:** For every stuck order, the worker actively calls the Stripe API (`GET /v1/payment_intents/{id}`) to ask for the absolute truth.
4. **The Resolution:**
   - If Stripe says "Failed," the worker marks the DB as `FAILED` and releases the Redis lock.
   - If Stripe says "Succeeded," the worker updates the DB to `SUCCEEDED` and secures the ticket.

---

## 4. The "Expired Lock" Race Condition (The Saga Pattern)

**The Scenario:** Time runs out. The user takes too long to pay, or the network outage lasts longer than the Redis TTL.

### The Timeline

| Time | Event |
|------|-------|
| **Minute 0** | User A locks the seat. (Redis TTL: 10 mins). |
| **Minute 9:30** | User A pays. DB marks order `PENDING`. Network dies. Webhook is lost. |
| **Minute 10:00** | Redis Lock expires. Seat becomes available. User B buys it instantly. |
| **Minute 12:00** | The Reconciliation Cron Job wakes up. It asks Stripe about User A's order. Stripe says, "Yes, we took their $20 at minute 9:30." |

**The Conflict:** The system has taken User A's money, but User B now owns the seat.

### The Solution: Compensating Transactions

Because we cannot use database locks across external API boundaries, we must use the **Saga Pattern**. Since the system cannot fulfill the order, it must execute a new, reversing action to return to a consistent state.

1. **The Action:** The Cron Job sees that the payment succeeded but the fulfillment failed (seat is gone).
2. **The Reversal:** The Cron Job makes an API call to Stripe: `POST /v1/refunds` passing the Idempotency Key.
3. **The State Update:** Stripe refunds the user. The Cron Job updates the database Order status to `FAILED_AND_REFUNDED`.
4. **The UX:** The user receives a notification: *"Your reservation expired before payment completed. You have been fully refunded."*
